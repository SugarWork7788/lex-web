## Dimension 7 — Risks

### Warnings (non-blocking)

**W-1 — Static aliasing fixture goes stale (acknowledged, but worth re-flagging at execute time)**
- The `must_NOT_contain_idmats` set in `EXPECTED_ISSUE_2025_100` is the 2026/42 idMat snapshot from RESEARCH date 2026-05-11. As soon as a new issue is published (regular cadence ~weekly), this set becomes the formerly-most-recent set, and the live aliasing-bait `_fetch_aliasing_bait` will use a different set. The plan documents this in the fixture comment ("STATIC defense; the production scraper ALSO refreshes the live aliasing-bait set on startup"), AND the plan does refresh `aliasing_bait` per scraper run. So production is safe. The TEST, however, will keep asserting against the stale 2026/42 set forever until the fixture is re-pinned. Today's date is also 2026-05-11 so the fixture is fresh — no immediate issue. Future re-runs of the test suite (weeks later) may need the fixture re-pinned if dv.parliament.bg reuses idMats… which RESEARCH says they don't (sequential). So this is documentation-only worth flagging, not a real bug.
- Severity: WARNING. Recommendation: leave as-is; the inline comment + the live `_fetch_aliasing_bait` redundancy is sufficient.

**W-2 — Smoke gate semantic check vs alias regression**
- The historical-issue smoke (`test_2025_100_official_toc_count_live`) asserts (a) ≥12 idMats present, (b) sample idMats ⊂ fetched, (c) no overlap with the static must_NOT_contain set. This catches the precise alias-to-2026/42 regression. **However**, if the upstream regresses to alias 2025/100 to a DIFFERENT issue (e.g., 2025/95 or 2026/40), the smoke would still see ≥12 idMats, the sample 238979/239155 might not be present (test would catch it via the sample-subset assertion), but the test would NOT actively detect aliasing — it would just fail the subset assertion with a confusing error. The defense is real (sample-subset assertion catches arbitrary aliasing), but the failure message would not say "aliasing." The Task 3.5 step 4 (`grep -oE "брой:\s*[0-9]+"` on idMat 238979's per-act page) is the unambiguous check. Acceptable.
- Severity: WARNING. Diagnostic-clarity nit, not a coverage gap.

**W-3 — Backfill completion not gated by checkpoint (per CONTEXT D-05)**
- Task 4 step 4 only requires "5 minutes of progress + zero aliasing events" before approval. The full 2-3 h backfill completes after the checkpoint releases. CONTEXT D-05 explicitly authorizes this ("validation batch is unnecessary; just go"). The SUMMARY/ROADMAP markup happens with the backfill still possibly running — but this is acceptable per CONTEXT and the plan §G + §<output>.4 documents the post-merge follow-up convention. Accepted with explicit note.
- Severity: WARNING (documentation only). No fix needed; CONTEXT explicitly authorizes the pattern.
