## Deviations from plan

1. **Task 3 frontmatter task_count was 4; actual count was 5** — corrected pre-commit. Off-by-one in planner's frontmatter (Task 3.5 was a separate `<task>` block but missed in the count).
2. **Three runtime bugs caught by the smoke loop, not by RESEARCH** — RESEARCH probed each URL with curl in isolation, missing three issues that only surface in a long-lived shared httpx session:
   - **Bug A — `_IDMAT_RE` strict pattern.** Stateless GETs (no cookie) trigger server-side `;jsessionid=...` URL injection. Phase 8's regex required `showMaterialDV.jsp\?idMat=` immediately adjacent. Fix: relax to `showMaterialDV\.jsp(?:;jsessionid=[A-F0-9]+)?\?idMat=`.
   - **Bug B — title-extraction structure differs between `index.faces` and `materiali.faces`.** index.faces nested everything in one `<td>` (Phase 8 Path A); materiali.faces puts each anchor in its own tiny `<table>` with the title as a TEXT NODE between `</strong>` and that table. Fix: added Path B fallback in `_extract_title_for_anchor` that walks `previous_siblings` of the parent `<table>`.
   - **Bug C — sticky-session aliasing under shared cookie jar.** Calling razdel_=2 right after razdel_=1 on the SAME `httpx.Client` returns razdel_=1's TOC again. Cookie-based session state. Fix: `_fetch_issue_toc` uses an ephemeral fresh `httpx.Client` per call. Cost: ~5-10 ms per fetch (one extra TCP handshake) — negligible vs the 1.5 s polite delay. Regression test `test_2025_100_sticky_session_regression_live` makes two calls on a shared outer client and asserts the result sets are disjoint.

   These three runtime bugs vindicate the plan's BLOCKING checkpoint design — they would have shipped silently to a "PR merged → backfill launched → 0 net rows again" loop without the manual smoke gate.

3. **Phase 1 follow-up commits in lex-brain `main`** — already merged (pushed during Phase 8 ship), so Phase 8.1 branched cleanly off `main` with no carrying-forward concern.
