# Phase 8.1 — DV scraper navigation hotfix — Research

## Summary

The Phase 8 scraper used the wrong navigation strategy. It assumed JSF POST-back over `index.faces` could be parameterized to fetch any historical issue's TOC, but **the `index.faces` JSF dataTable always responds with the most-recent issue's TOC regardless of `idObj`/`broi_`/`date_izd_` POST params**, and `broi_form:next_` pagination on `index.faces` likewise always falls back to the index/most-recent issue (`active_tab=1` in response). Both bugs reproduced cleanly during this research session.

**The fix:** Two new verified endpoints replace the broken postback walk:
1. **`GET https://dv.parliament.bg/DVWeb/materiali.faces?idObj=N&razdel_=R`** — returns one issue's TOC. Stateless, no cookies needed, no ViewState. R=1 for official, R=2 for unofficial. Verified per-issue accuracy by fetching idMats and confirming each individual act's `брой:` header matches the requested issue.
2. **`POST https://dv.parliament.bg/DVWeb/broeveList.faces`** — direct archive listing (NOT routed through tab2 of index.faces). With `broi_form:from_date` + `broi_form:to_date` POSTed via the `btnFind11` image submit, returns a paginated list of issues in the date range. `broi_form:next_` pagination works correctly here (returns `active_tab=2` instead of bouncing to 1). Per-issue-number filter via `broi_form:_idJsp67=N` also works (filters across all years where that broi number exists).

**Earlier-session memory observation S994/S997-S999 was partially right and partially wrong.** The "stateless GET path" claim is correct for `materiali.faces?idObj=N` (per-issue TOC) and `showMaterialDV.jsp?idMat=N` (per-act body), but the original Phase 8 scraper went the wrong way: it tried to drive these through the `index.faces` POST-back graph rather than calling the stateless endpoints directly. The new architecture sidesteps the broken `index.faces` postback layer entirely.

**Primary recommendation:** Replace `_post_tab2`/`_post_issue_toc`/`_post_pagination` with: (a) a small POST helper against `broeveList.faces` for issue enumeration with date filter + pagination, (b) a stateless `GET materiali.faces?idObj=N&razdel_=R` for per-issue TOC. Keep all parsing helpers (`_parse_issue_rows`, `_parse_toc_acts`, `_extract_title_for_anchor`, `infer_act_type`, `strip_jsessionid`) — the response shapes are byte-identical to what Phase 8 already parses, just delivered via different endpoints.

### Q1 — Stateless URL discovery [VERIFIED]

**Answer:** Three endpoints, all live-probed:

1. **Per-issue TOC (the missing piece):**
   ```
   GET https://dv.parliament.bg/DVWeb/materiali.faces?idObj=<N>&razdel_=<1|2>
   ```
   - Stateless: no cookie, no session, no ViewState
   - `idObj` = the per-issue identifier captured from the archive listing
   - `razdel_=1` returns the official section's TOC; `razdel_=2` returns the unofficial section's TOC
   - Returns ~24-30 KB HTML containing `<a href="showMaterialDV.jsp?idMat=N">…Преглед на материала</a>` anchors with the same row shape that Phase 8's `_parse_toc_acts` + `_extract_title_for_anchor` already handle
   - **Important quirk:** the page header text "брой: 46, от дата 11.5.2026" is a **dynamic but misleading template**. It always shows what looks like a current/projected issue label regardless of which idObj was requested. The actual TOC content is correctly per-issue. Verification approach: parse idMats from the response, then confirm via per-act `showMaterialDV.jsp?idMat=N` page where the `брой:` line IS accurate.

2. **Issue enumeration (the second missing piece):**
   ```
   POST https://dv.parliament.bg/DVWeb/broeveList.faces
   Content-Type: application/x-www-form-urlencoded

   broi_form_SUBMIT=1
   broi_form:btnFind11.x=15
   broi_form:btnFind11.y=10
   broi_form:from_date=DD.MM.YYYY    # Bulgarian-locale format, NOT ISO
   broi_form:to_date=DD.MM.YYYY
   broi_form:not_first=1
   active_tab=2
   javax.faces.ViewState=<from previous response>
   ```
   - **Crucially: NOT `index.faces`.** POSTing the same body to `index.faces` returns `active_tab=1` (bounces back to most-recent-issue index). Posting to `broeveList.faces` correctly stays on `active_tab=2` and returns the filtered listing.
   - First call: GET `https://dv.parliament.bg/DVWeb/broeveList.faces` to capture the initial ViewState (no auth/session needed)
   - Returns up to 10 issue rows per page, with each row encoding `[broi_, idObj, date_izd_, razdel_]` in an `oamSubmitForm()` onclick string — Phase 8's `_parse_issue_rows` regex already handles this exact shape

3. **Pagination (works on broeveList.faces, broken on index.faces):**
   ```
   POST https://dv.parliament.bg/DVWeb/broeveList.faces

   broi_form_SUBMIT=1
   broi_form:_idcl=broi_form:next_
   broi_form:_link_hidden_=broi_form:next_
   broi_form:from_date=DD.MM.YYYY    # echo the same filter
   broi_form:to_date=DD.MM.YYYY
   broi_form:not_first=1
   active_tab=2
   javax.faces.ViewState=<from previous filtered response>
   ```
   Returns the next 10 issues. Verified by walking pages 1-3 of 2024: issues 110-101, 100-91, 90-81 — sequential, no duplicates, no aliasing.

4. **Per-act body (Phase 8 already had this right — unchanged):**
   ```
   GET https://dv.parliament.bg/DVWeb/showMaterialDV.jsp?idMat=<N>
   ```
   Confirmed stateless and unchanged. Strip `;jsessionid=...` from `resp.url` before persisting.

### Q2 — Issue enumeration without pagination [VERIFIED]

**Answer:** Pagination works on `broeveList.faces` — but the `index.faces` postback pagination is broken. Use the working endpoint.

**Two complementary enumeration strategies:**

**Strategy A: Date-range walk (recommended for backfill).**
- POST `broeveList.faces` with `from_date=01.01.2024` + `to_date=31.12.2024` → 110 results in 2024 (verified)
- Walk pages with `broi_form:next_` POST until results exhausted
- For 2-year backfill (2024 + 2025 + 2026-to-date): 3 separate date-filtered walks
- 2024 = 110 issues = 11 pages; 2025 = ~115 issues = 12 pages; 2026-to-date = 42 issues = 5 pages → **~28 listing-page fetches** for full backfill (negligible vs the ~10,000 act-body fetches that dominate runtime)
- **Year transition seamless** (Q5 below): a date range that crosses YYYY-01-01 boundary returns issues from both years in a single response, ordered by date desc

**Strategy B: Direct broi-number filter (works for smoke / spot-check).**
- POST `broeveList.faces` with `broi_form:_idJsp67=100` → returns issue 100 from EVERY year that has one (36 results spanning 1989-2025; verified)
- Useful for the smoke fixture (Q3) and any one-off "where is issue X of year Y?" lookup

**Resumability path** (matches D-04):
1. After backfill walk completes, re-query `dv_issues LEFT JOIN dv_acts HAVING COUNT(a.id)=0` to find empty rows
2. For each empty row, parse `idObj` from `source_url` (already stored — verified live)
3. Fetch its TOC via `GET materiali.faces?idObj=N&razdel_=1` and `&razdel_=2`
4. Insert acts via `INSERT ... ON CONFLICT (source_url) DO NOTHING`

### Q4 — Refetch predicate [VERIFIED]

**Answer:** **`LEFT JOIN dv_acts ... GROUP BY i.id HAVING COUNT(a.id) = 0`.** Live-DB EXPLAIN ANALYZE confirms 0.115 ms execution time at current scale (10 issues, 10 acts).

```sql
SELECT i.id, i.year, i.issue_number, i.source_url
FROM dv_issues i
LEFT JOIN dv_acts a ON a.issue_id = i.id
GROUP BY i.id
HAVING COUNT(a.id) = 0;
```

**Live-probe result (2026-05-11):**
- Returns 9 rows: issues 33-41 of 2026, all with `acts=0` — exactly the corrupt-shell rows from the Phase 8 production run
- Issue 42 of 2026 (acts=10) correctly excluded
- Plan: `HashAggregate → Hash Right Join → Seq Scan dv_acts + Seq Scan dv_issues` → at 4,107 issues × ~30 acts ≈ ~120K acts the seqscans get costly. **Mitigation:** the live schema needs `CREATE INDEX IF NOT EXISTS dv_acts_issue_id_idx ON dv_acts(issue_id)` to convert the right-side scan to an index-only scan. The Phase 8 schema migration did NOT add this index automatically (Postgres FKs don't auto-index the referencing column). **Action for the planner:** include this index in the migration, or verify it via `SELECT * FROM pg_indexes WHERE tablename='dv_acts' AND indexdef LIKE '%issue_id%'` before relying on the predicate at scale.

**No need for `expected_act_count` precomputation** — the predicate `COUNT(a.id) = 0` correctly identifies "TOC was never fetched" since both razdels are required to populate any rows. (Edge case: if razdel=1 succeeds and razdel=2 fails, the issue has SOME acts and the predicate marks it complete, missing the unofficial section. Mitigation: per-issue handler MUST insert all razdel=1 + razdel=2 acts atomically per issue, OR the predicate must be tightened to `COUNT(a.id) < expected`. The simpler atomic-insert-per-issue approach is preferred.)

## Recommended scrape_dv.py architecture

The new walk shape is structurally simpler than Phase 8's. Three key replacements; everything else (parsing helpers, DB upsert helpers, polite delay, logging, jsessionid stripping, act-type inference, http_retry wrapper) carries forward unchanged.

**Replacement 1 — Issue enumeration:** A single new helper `_enumerate_issues_for_year(year: int) -> Iterator[IssueRow]` POSTs the date-filtered query against `broeveList.faces` (`from_date=01.01.YYYY`, `to_date=31.12.YYYY`), parses the rows via the existing `_parse_issue_rows` (zero changes), and walks pagination via `broi_form:next_` POST until `Намерени резултати: N` is exhausted (track count: yield up to N rows then stop). Per-page logging structured event `enum_page_done` with `{year, page, broi_first, broi_last, rows}`. Three calls per backfill: 2024, 2025, 2026. Bounded by `BACKFILL_FLOOR = date(2024, 1, 1)`.

**Replacement 2 — Per-issue TOC:** A stateless GET helper `_fetch_issue_toc(client: httpx.Client, id_obj: str, razdel: int) -> str` calling `GET https://dv.parliament.bg/DVWeb/materiali.faces?idObj={id_obj}&razdel_={razdel}` with no session reliance. Returns the HTML body for the existing `_parse_toc_acts` + `_extract_title_for_anchor` parsers (zero changes). Each issue triggers TWO calls (razdel=1, razdel=2) before moving to the next issue — preserves Phase 8's dual-razdel invariant.

**Replacement 3 — Per-issue handler with view-aliasing guard (D-03):** Wrap `_process_issue` with a defense-in-depth check: BEFORE inserting acts, compare the fetched razdel=1 idMat set to the most-recent issue's known idMat set (cached on first call). If the sets are identical (catastrophe — server is aliasing again), log structured event `view_aliasing_detected` with `{requested_id_obj, requested_broi, fetched_idmats}` and skip insert (don't corrupt DB). Cheap because the most-recent set is fetched once per scraper run.

**Removed:** `_post_tab2`, `_post_issue_toc`, `_post_pagination`, `_get_index` (the index-page session bootstrap is no longer required since both new endpoints work statelessly or via fresh-session POST). The `ViewExpiredError` class can stay but will fire much less often (only inside the `broeveList.faces` paginated walk; per-issue and per-act fetches are stateless).

**Resumability flow rewritten:**
1. At scraper start: query `dv_issues LEFT JOIN dv_acts HAVING COUNT(a.id)=0` → list of `(year, broi_, source_url)` to refetch
2. For each refetch row: parse `idObj` from `source_url` regex (already in `dv_jsf.py` style — small new helper `parse_id_obj_from_source_url(url) -> str`), then call `_fetch_issue_toc(client, id_obj, razdel)` for both razdels, insert acts via existing `_upsert_act`
3. Then proceed to fresh enumeration walk for `(year, broi_)` pairs not yet in `dv_issues`

## Pitfalls

1. **The `materiali.faces` page header is a misleading template.** The text "брой: 46, от дата 11.5.2026" appears regardless of which idObj is requested — appears to be the site's running "current/projected next issue" header. **Do NOT validate the per-issue TOC by checking the page header.** Validate by fetching one of the parsed idMats and checking `showMaterialDV.jsp?idMat=N`'s `брой:` header instead (verified: per-act header IS accurate).

2. **Bulgarian date format on `broeveList.faces` form.** The `from_date` / `to_date` text inputs accept **`DD.MM.YYYY`** (Bulgarian locale), NOT ISO `YYYY-MM-DD`. Verified by submitting `01.12.2025` + `31.12.2025` → 13 results in Dec 2025; submitting ISO format silently returned 0 filter effect (returned page 1 unfiltered). Format the date strings explicitly: `f"{d.day:02d}.{d.month:02d}.{d.year}"`.

3. **`broeveList.faces` requires its OWN ViewState.** Each fresh GET to `broeveList.faces` returns a base64 ViewState that decodes to `/broeveList.jsp` (vs `index.faces`'s ViewState which decodes to `/index.jsp`). They're not interchangeable. The scraper MUST do GET `broeveList.faces` first to capture the listing-form ViewState, then POST against `broeveList.faces` (NOT `index.faces`).

4. **Image-submit button needs `.x` and `.y` coordinates.** `broi_form:btnFind11` is `<input type="image">`. The browser submits `name.x=N&name.y=N` for the click coordinates. Some servers accept just `name=` but **dv.parliament.bg requires both `.x` and `.y`** — without them the POST is ignored and the response defaults to page 1 of the unfiltered listing. Send `broi_form:btnFind11.x=15&broi_form:btnFind11.y=10` (any values work; matters that the keys are present).

5. **Pagination POST MUST echo the original date filter.** `broi_form:next_` works only if the same `from_date`/`to_date` are POSTed alongside. Without them the server resets the filter and returns to page 1 of unfiltered. Treat the filter params as part of the JSF "form state" and re-send on every paginated POST.

6. **Sparse, irregular `idObj` space.** Probes at `idObj=12200, 12400, 12450, 12455-7` all returned HTTP 500 (gaps in the ID sequence). Brute-force scanning `idObj` is wasteful — many requests get 500s. **Always enumerate via `broeveList.faces` first** to get the live (broi, idObj) pairs; never guess.

7. **`materiali.faces?idObj=N` does NOT include razdel=2 in the same response.** A separate GET with `&razdel_=2` is required. Easy to miss — assert per-issue act counts with both calls combined. (Same dual-fetch invariant as Phase 8.)

8. **`materiali.faces` URLs in `<a href>` carry `;jsessionid=...`.** Even though the GET is "stateless," the HTML it returns contains anchor tags with `;jsessionid=...` segments stamped into every `showMaterialDV.jsp` href. The existing `strip_jsessionid()` helper handles this for the persisted source_url; just make sure it runs on every URL extracted from the HTML before saving.

9. **Phase 8's title-extraction bug applies to the new path too.** The `<a href="showMaterialDV.jsp">` link text on `materiali.faces` is the same `Преглед на материала` ("View the material") it was on `index.faces`. The `_extract_title_for_anchor()` helper (added mid-Plan 8-01 per 08-01-SUMMARY.md §"Deviations from plan") MUST be used unchanged — verified by inspecting the new TOC pages have the same `<td>` row shape with `<strong>{issuing_body}</strong><br>{title}<br>...стр. N<br><a>...Преглед на материала</a>`.

10. **`dv_acts.issue_id` FK index is NOT auto-created.** Postgres only auto-indexes PRIMARY KEY columns, not FOREIGN KEY referencing columns. The Phase 8 schema migration `db/dv_schema.sql` did not add an index on `dv_acts.issue_id`. The refetch predicate's LEFT JOIN currently uses Seq Scan on `dv_acts` — at 120K rows this becomes slow. **Planner should add `CREATE INDEX IF NOT EXISTS dv_acts_issue_id_idx ON dv_acts(issue_id)`** to a new migration file (idempotent, additive, mirrors the Phase 8 migration pattern). Skipping this is acceptable for the immediate hotfix (current 10-row DB is fine), but the backfill stresses the predicate hard at the 9-empty-row refetch step + every subsequent resumability check.

11. **`broeveList.faces` post-back to `next_` advances the LISTING, not a TOC.** Don't confuse it with `index.faces`'s pagination (which is broken). They look syntactically similar (both POST `broi_form:_idcl=broi_form:next_`) but the endpoint matters.

**Caveats:**
- The act count of 12+30=42 reflects the issue's content as of 2026-05-11. If dv.parliament.bg ever republishes/amends an act, the count could shift. The scraper test should hard-assert the current numbers; if the fixture diverges, treat as either a real upstream change (re-pin the test) or a scraper regression (investigate). 6-month-stale issues are extremely unlikely to change.
- The "must_NOT_contain_idmats" list is the most-recent issue's idMat set as of 2026-05-11. When the gazette publishes a new most-recent issue, the test's defense-against-aliasing list will become stale. Recommended: in the per-issue handler (D-03 implementation), fetch the live most-recent-issue idMat set at scraper start (one cheap GET to `materiali.faces?idObj=12499`), cache it, and assert against the LIVE set in addition to the static smoke fixture set. This way the production guard auto-updates.
