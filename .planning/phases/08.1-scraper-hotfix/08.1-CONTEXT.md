# Phase 8.1 — DV scraper navigation hotfix

## Background (what failed)

After Phase 8 PR #6 merged (lex-web) + PR #7 merged (lex-brain), the operator launched the documented full 2-year backfill (`nohup uv run python scripts/scrape_dv.py …`). The scraper terminated after ~7 minutes with **0 net new rows**:

```
{"event": "page_empty", "page": 2}
{"event": "walk_complete", "pages": 2, "issues": 10, "total_acts": 0}
```

Two JSF protocol bugs were aliased to "right answer" by the most-recent-issue smoke gate (Бр.42 = newest), so they slipped through verification:

1. **`idObj` POST parameter is unreliable** — historical issue requests return the most-recent issue's TOC regardless of params. Documented in earlier-session research observations S996/S997: "all tested values return most recent issue regardless of input."
2. **Pagination POST-back returns empty page 2** — the `broi_form:next_` link doesn't actually advance the listing in our session.

**Live-DB damage:** 10 issue rows were inserted into `dv_issues` (Бр.33–42 of 2026). Only Бр.42 has acts (10) — the other 9 are empty shells because their TOC fetches all returned issue 42's content, which got deduped by `source_url` uniqueness and collapsed onto issue 42's existing rows.

The scraper IS the only reachable path to historical content via the published JSF surface. The bug class can ONLY be exercised against a known-historical issue, never the most-recent one.

## Phase scope (locked)

Minimal navigation rewrite:
1. Replace JSF POST-back per-issue + pagination navigation with the stateless GET path documented in earlier-session research observations S994 / S997 / S998 / S999 (memory: "Direct GET path discovered for per-issue act enumeration bypassing JSF postback maze"; "State Gazette endpoints fully stateless: showMaterialDV.jsp and materiali.faces work without cookies or sessions").
2. Add ONE new smoke gate against a known-historical issue (≥6 months old, NOT current year's most-recent) — the bug class can ONLY be caught against historical content.
3. Re-run the full 2-year backfill from scratch after the fix lands. Existing `(year, issue_number, issue_supplement)` UNIQUE constraint + the keep-and-update strategy (D-04 below) repairs the 9 corrupt empty-shell issues on second pass.

Everything else from Phase 8 stays as-is: dv_jsf.py helpers, DB schema, lex-web `/dv` UI, summarize endpoint, polite delay, structured logging shape, identifying UA, jsessionid stripping.

## Decisions

### D-01 — Navigation strategy: stateless GET, no JSF POST-back
- **What:** Drop `_post_tab2`, `_post_issue_toc`, `_post_pagination`, `_post_oamSubmitForm`-style helpers from `scrape_dv.py`. Replace with stateless `httpx.get()` calls. The `materiali.faces?idObj=N` endpoint and `showMaterialDV.jsp?idMat=N` endpoint both work without cookies or ViewState (verified in earlier-session research, S999).
- **Why:** The `idObj` POST parameter is unreliable (S996); the `broi_form:next_` pagination POST returns empty (live evidence). The stateless GET path is the only verified path to historical content.
- **Research must answer:** (a) Exact GET URL shape for per-issue TOC enumeration. (b) How to enumerate issue rows for a given year — direct `broi_=N&year=YYYY` GET? Year-summary page? (c) Whether `materiali.faces?idObj=N` returns the per-issue TOC in the same shape that the current `_parse_toc_acts` consumes (so the parser stays untouched), or whether the parser needs adaptation.

### D-02 — Smoke gate: pick a known-historical issue, NOT the most-recent
- **What:** Add a pytest fixture / integration smoke that ingests one specific historical issue (target: 2025 issue with documented act count), then asserts: (a) that issue's act_count matches expected, AND (b) the act idMats do NOT match the most-recent issue's idMats (catches the alias-to-newest regression).
- **Why:** The whole reason this bug shipped is that the smoke target was the newest issue; both bugs aliased to "looks correct." A historical-issue smoke catches both the idObj-aliasing AND any future variant.
- **Research must pick:** A specific issue (e.g., 2025/100) with stable, manually-verifiable act count. Document the expected count + a sample idMat from that issue's TOC in the test fixture.

### D-03 — Re-fetch invariant guard
- **What:** Per-issue handler asserts: every fetched TOC's idMats are NOT a strict subset of the most-recent issue's idMats. If they are, log a `view_aliasing_detected` structured event + skip the row (don't insert acts under the wrong issue_id).
- **Why:** Even after the navigation rewrite, the scraper should fail loudly if the upstream changes shape again. Cheap defense-in-depth.

### D-04 — Corrupt data: keep, UPDATE on second pass (not DELETE)
- **What:** Leave the 9 empty `dv_issues` rows (Бр.33–41 of 2026) in place. The new scraper detects "issue row exists with 0 acts" and refetches its TOC, producing the actual acts. Acts insert via the `(source_url)` UNIQUE constraint with `ON CONFLICT DO NOTHING`; new ones land normally.
- **Why:** The `(year, issue_number, issue_supplement)` UNIQUE constraint already protects against double-insert. UPDATE-on-second-pass is the same pattern Phase 8's resumability story relies on. DELETE-then-re-run loses the dv_issues.id UUIDs, which would be problematic if any FK refs existed (none yet, but future-proof).
- **Research must answer:** What's the exact predicate for "this issue row needs its acts refetched"? `LEFT JOIN dv_acts WITH COUNT(a.id)=0`? Or smarter (e.g., compare expected-act-count from TOC vs stored-act-count)?

### D-05 — Backfill restart: full 2-year, immediately after fix lands
- **What:** Once Phase 8.1 PR merges, kick off the full backfill (~10,000 rows × 1.5 s polite delay ≈ 2–3 h). Resumable; safe to leave running unattended.
- **Why:** Validation batch is unnecessary — the historical-issue smoke gate (D-02) already proves the navigation works against historical content; running a "small batch" is the same code path executed for slightly less time. Just go.

### D-06 — All Phase 8 invariants preserved unchanged
- Polite delay ≥ 1.5 s ± 0.5 s (D-02 from Phase 8 CONTEXT)
- Identifying User-Agent `lex-brain-scraper/1.0 (+https://lex-web-eta.vercel.app; ...)` (D-03 from Phase 8 CONTEXT)
- `strip_jsessionid()` on every persisted source_url (D-05 from Phase 8 CONTEXT)
- `INSERT ... ON CONFLICT DO NOTHING` for both issues + acts
- Structured JSON log per fetched issue + per fetched act (event=`act_done` / `issue_done`)
- `http_retry.py` byte-identical (Phase 1 D-12 — append-only convention)
- Both `razdel_=1` (official) AND `razdel_=2` (unofficial) sections fetched per issue (Phase 8 RESEARCH Q2)

## Open questions (for research)

1. **Q1 — Stateless URL discovery:** What's the canonical GET URL pattern for (a) per-issue TOC by `broi_`+`year` (without idObj), (b) year-summary listing of all issues, and (c) per-act body? `showMaterialDV.jsp?idMat=N` is known to work for (c). Memory observation S994 says (a)/(b) exist but not the exact URL — research must verify against live site.
2. **Q2 — Issue enumeration without pagination:** With ~250 issues/year × 2 years (2024 + 2025 + partial 2026), can we enumerate by direct `broi_=N&year=YYYY` GET requests, or do we need an index page? The current pagination POST-back is broken; we need an alternative.
3. **Q3 — Smoke fixture choice:** Pick a specific historical issue with verified act count. Suggested: 2025/100 (mid-year, well-defined edition) — but research should confirm by fetching its TOC manually and counting.
4. **Q4 — Refetch predicate:** What's the cheapest SQL probe that identifies "issue row needs its acts" — `LEFT JOIN COUNT(a.id)=0`, or precompute `expected_act_count` at insert time and compare?
5. **Q5 — Year transition:** When the scraper crosses the 2025→2024 year boundary while iterating broi_ ranges, does it need a different URL pattern? Some JSP archives partition by year-page.
