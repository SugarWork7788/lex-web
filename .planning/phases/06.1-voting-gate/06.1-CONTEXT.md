# Phase 6.1 — Voting gate

## Phase scope (locked from ROADMAP)

Gate `/audit` voting: anonymous users can still read findings (the `/audit` route stays public), but the vote button no longer accepts anonymous votes. Authenticated users vote and each vote is attributed to `auth.users.id` in `audit_votes` alongside the existing IP+fingerprint anti-abuse columns. Historical anonymous votes are preserved.

**Requirements:** AUTH-08, AUTH-10 (AUTH-09 already satisfied by Phase 5 proxy; AUTH-11 already satisfied by Phase 4 `/profile` page — see ROADMAP Coverage table).

**Cross-phase note:** Phase 5 (auth proxy + `requireAuth()`) is the dependency. The `/audit` route is NOT in the proxy matcher (anonymous reading must stay open) — the gate is enforced inside the vote button + vote API only.

## Decisions

### D-01 — Vote button anonymous UX = disabled button + side "Влез за глас →" link

For anonymous users on `/audit/finding/[id]`, keep the `👍 Подкрепи` button visually present (greyed-out / `disabled` attribute), with a small `Влез за глас →` link rendered immediately to the right of the count. The disabled button gets a `title` tooltip explaining the gate.

```
[ 👍 Подкрепи ]  142 гласа · Влез за глас →
```

The link target is `/sign-in?returnTo=/audit/finding/[id]` (URL-encoded). The `returnTo` query param is the **current page's path including hash**, so after sign-in the user lands back where they were ready to vote.

**Why this option (not link-as-button replacement, not inline prompt panel):**
- Visual continuity — authed users see the same control they'd see after signing in, so the layout doesn't shift when they auth.
- Tap-target compliance — the link is a real `<a>` (mobile-friendly), not a click handler that redirects.
- Anti-pattern avoided — we don't hide the button (which would make the gate invisible until hover), and we don't replace the button with a CTA (which costs visual continuity).

**What this means for plan 06.1-01:**
- `<VoteButton>` becomes a Server Component (or accepts a `user: User | null` prop from a Server Component parent) so the anon/authed split is decided server-side, not via a client `useSession()` hook that would flash anonymous UI for half a second on hydration.
- Keep the existing fingerprint() helper for the authed code path? **No** — D-03 below makes fingerprint a forensic-only column the client no longer computes. The `<VoteButton>` client portion shrinks.

### D-02 — Preserve historical anonymous votes; user_id stays NULL on existing rows

The migration adds `audit_votes.user_id uuid REFERENCES auth.users(id) ON DELETE SET NULL` as **nullable**. Historical anonymous votes keep `user_id IS NULL` and continue counting toward `findings.vote_count`. Going forward, new votes (now required to be authed) get a non-NULL `user_id`.

**Why this option (not wipe, not exclude-from-count):**
- Wiping deletes existing engagement signal (the audit findings already have months of accumulated votes).
- Exclude-from-count would silently drop all current counts on deploy — visible regression for anyone watching.
- Preserve-with-NULL is the only option that ships with zero visible count change.

**What this means for plan 06.1-01:**
- DB migration is purely additive: `ALTER TABLE audit_votes ADD COLUMN user_id uuid REFERENCES auth.users(id) ON DELETE SET NULL` (nullable, no default).
- No `UPDATE` statement on existing rows.
- No backfill needed; no `findings.vote_count` recompute.
- The migration is idempotent (`ADD COLUMN IF NOT EXISTS`).

### D-03 — Uniqueness: `UNIQUE(finding_id, user_id)`; `ip_hash` is rate-limit-only; `fingerprint_hash` is forensic-only

Going forward, vote uniqueness is keyed on `(finding_id, user_id)` — one vote per user per finding. The existing `ip_hash` column stays and continues to power the 50-votes-per-IP-per-24h global rate limit (defense against compromised-account swarms). The `fingerprint_hash` column stays in the schema for forensic audit but is **no longer queried in the hot path** and is **no longer sent by the client** for new votes.

**Why this option (not drop-fingerprint, not belt-and-suspenders):**
- `user_id` is the real identity once auth is required — fingerprint is redundant for uniqueness.
- Dropping fingerprint entirely loses a useful forensic signal (e.g., "the same browser fingerprint voted on 50 findings across 50 throwaway accounts in 5 minutes" is still a smell worth being able to investigate post-hoc).
- Belt-and-suspenders rejects votes from households sharing IPs (parent + child both signed into separate accounts on the same WiFi) — over-conservative for the audit use case.

**What this means for plan 06.1-01:**
- Migration adds `CREATE UNIQUE INDEX IF NOT EXISTS audit_votes_user_finding_unique ON audit_votes (finding_id, user_id) WHERE user_id IS NOT NULL` — partial-unique so historical NULL rows aren't constrained.
- **Migration must DROP the existing `UNIQUE(finding_id, ip_hash)` constraint** (`audit_votes_finding_id_ip_hash_key`) — otherwise two users sharing a household IP both can't vote on the same finding (rejected in D-03 option C).
- **Migration must DROP the existing `UNIQUE(finding_id, fingerprint_hash)` constraint** (`audit_votes_finding_id_fingerprint_hash_key`) and `ALTER COLUMN fingerprint_hash DROP NOT NULL` — otherwise we can't insert rows without a client-supplied fingerprint.
- **`ip_hash` stays NOT NULL** — always available from request headers, used as rate-limit key (50/24h via `audit_votes_ip` btree index, already present).
- `/api/audit/vote` removes the `fingerprint` field from the request body schema and stops calling the client-side `fingerprint()` helper.
- `/api/audit/vote` still records `ip_hash` and still enforces the 50/24h cap. Per-finding "already voted?" check becomes `SELECT 1 FROM audit_votes WHERE finding_id = $1 AND user_id = $2`.
- The existing `fingerprint_hash` column stays in the schema (no DROP COLUMN) — historical rows keep their hashes; new rows write NULL.
- The `<VoteButton>` client component drops the `fingerprint()` function and the SHA-256 subtle-crypto call — meaningful client-side simplification.

**Live-DB schema audit (2026-05-12 via psycopg2):**
```
audit_votes
  id               uuid NOT NULL default gen_random_uuid() PK
  finding_id       uuid NOT NULL REFERENCES audit_findings(id) ON DELETE CASCADE
  ip_hash          text NOT NULL
  fingerprint_hash text NOT NULL  ← drop NOT NULL in 06.1-01
  voted_at         timestamptz default now()

Constraints to DROP:
  audit_votes_finding_id_ip_hash_key          UNIQUE(finding_id, ip_hash)
  audit_votes_finding_id_fingerprint_hash_key UNIQUE(finding_id, fingerprint_hash)

Indexes to KEEP:
  audit_votes_pkey       (id)
  audit_votes_finding    (finding_id) — supports the increment_audit_vote RPC
  audit_votes_ip         (ip_hash)     — supports the 50/24h rate-limit lookup

Row count: 2 — historical preservation per D-02 is essentially free.
```
**No SQL file under `db/` defines `audit_votes`** — the table predates the GSD planning era. The new `db/audit_votes_user_id_migration.sql` (or extension of an existing applier) will be the first repo-tracked DDL for this table.

### D-04 — `/api/audit/vote` returns 401 for anonymous; client never POSTs anon

`/api/audit/vote` calls `createRouteHandlerSupabase()` + `await supabase.auth.getUser()`. If null → return `Response.json({success: false, reason: "auth_required"}, {status: 401})`. The client `<VoteButton>` is server-rendered with `user` already known, so for anonymous users the POST handler is never invoked (the disabled button has no `onClick` that fetches). The 401 path is the belt-and-suspenders defense against direct API hits.

**What this means for plan 06.1-01:**
- Route handler gets a 3-line preamble: `const supabase = await createRouteHandlerSupabase(); const { data: { user } } = await supabase.auth.getUser(); if (!user) return Response.json(...401)`.
- No use of the existing `lib/rate-limit.ts` here — the 50/24h cap is already in the route via the `recent.count` check; we don't double-gate.
- Test fixture for the 401 path lives in `__tests__/audit-vote-route.test.ts` (new file, or extend an existing one if it exists).

## Anti-patterns to avoid

- **Don't gate `/audit/finding/[id]` itself.** Anonymous reading is explicitly required by the phase scope; the gate lives only in the button + the vote API.
- **Don't redirect anonymous POST to `/api/audit/vote` to `/sign-in`** — return 401 JSON. Anonymous users should never POST in the first place (D-04), but if they do (e.g., via curl or browser extension), the API returns a clean JSON error, not a 302.
- **Don't drop `fingerprint_hash` column.** Historical rows have it; downstream forensic queries may want it. Keep the column, stop querying it, stop writing to it for new rows.
- **Don't recompute `findings.vote_count`.** The atomic `increment_audit_vote` RPC continues to be the only writer; we add `user_id` as a new column on the insert and the RPC keeps incrementing.
